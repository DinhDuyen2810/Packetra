Capture Workflow Guide

This guide focuses on the capture lifecycle itself: selecting the correct interface, deciding whether to use a capture filter, running the capture, and preserving what you collected.

Back to Help Center

1. Choosing the correct interface

  • Choose the interface that shows active traffic in the interface selector instead of guessing by name.
  • On laptops and desktops, the correct interface is commonly the active Ethernet or Wi-Fi adapter.
  • VPN, virtual adapters, WSL adapters, and loopback-style interfaces may be valid but should only be chosen when you know your target traffic lives there.
  • For remote capture, the interface list is loaded from the remote host rather than the local machine.

2. When to use a capture filter

Use a capture filter when volume is high and you already know the traffic boundary you care about. If you are still exploring, capture more broadly and narrow down later with display filters.

host 192.168.1.10
port 443
tcp port 80
udp port 53
net 10.0.0.0/8
host 192.168.1.10 and port 3389
not broadcast and not multicast

These are BPF-style capture filters, not Packetra display filters.

3. Start, Stop, and Restart

  • Start begins a new live capture session.
  • Stop ends the active capture but keeps the data already collected in the current view.
  • Restart is the fast reset path when you want to discard the current live run and begin a new one immediately.

4. A reliable capture sequence

  1. Select the correct interface.
  2. Apply a capture filter only if you are confident about the traffic scope.
  3. Start capture.
  4. Trigger the system behavior you want to observe.
  5. Stop capture as soon as the relevant event has been captured.
  6. Use display filters for investigation.
  7. Save the capture if it has evidentiary or troubleshooting value.

5. Performance and quality tips

  • If the UI becomes busy, stop the live capture before applying many investigative filters.
  • If the capture is too large, use time-bounded runs and save multiple smaller files instead of one huge file.
  • If you are debugging a specific transaction, trigger only that transaction while capturing.
  • If you need both full context and narrow evidence, save the broader capture first, then work on filtered views.

6. Common problems during capture

  • No packets visible: most often the wrong interface was selected or the capture filter is too strict.
  • Packets appear but not the traffic you want: the host may be using another adapter, a VPN path, or IPv6 while you expected IPv4.
  • Local Windows capture does not start: verify Npcap status in the Help menu.
  • Remote Linux capture fails: verify SSH access and `tcpdump` availability on the remote host.
