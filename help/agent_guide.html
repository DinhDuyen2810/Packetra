Remote Capture Guide

This guide is intentionally Linux-first. It explains the actual remote capture workflow used by Packetra: SSH into the remote host, list interfaces, and stream packet data back via tcpdump.

1. How Packetra remote capture works

  • Packetra connects to the remote system over SSH using Paramiko.
  • For Linux hosts, Packetra lists interfaces with tcpdump -D.
  • When capture starts, Packetra runs a command in the form:
tcpdump -n -s 0 -i 'INTERFACE' -U -w -
  • The packet stream is written to standard output and consumed locally by Packetra.
  • If Packetra capture options disable promiscuous mode, -p is added.
  • If you enter a capture filter, Packetra appends that BPF expression to the remote tcpdump command.

4. If permissions are not enough

On many Linux systems, raw capture requires elevated privilege or packet-capture capability. If the SSH user cannot access interfaces directly, use one of these approaches:

  • Run the remote session as a user that already has capture rights.
  • Grant capability to tcpdump on the remote host if that matches your security policy.
  • Use a privileged user only if that is acceptable in your environment.
sudo setcap cap_net_raw,cap_net_admin=eip $(command -v tcpdump)

After changing capabilities, test again with:

tcpdump -D
tcpdump -n -s 0 -i YOUR_INTERFACE -c 5

5. Packetra setup steps for a Linux remote host

  1. Open Manage Interfaces.
  2. Go to the Remote Interfaces tab.
  3. Click Add to create a remote host row.
  4. Fill in:
    • Host / Device URL: the remote IP or hostname
    • Port: usually 22
    • OS: set to linux
    • Username: SSH login name
    • Auth Type: key/agent or password, depending on your setup
    • Password: only if password auth is used
  5. Select the row and click Connect & Load Interfaces.
  6. Tick the interfaces you want Packetra to show in the main interface picker.
  7. Confirm the dialog and start capture from the chosen remote interface.

6. Full window flow inside Packetra from first click to first remote capture

  1. Open the local interface/capture setup area and click Manage Interfaces.
  2. In Manage Interfaces, switch to the Remote Interfaces tab.
  3. Click Add to create a new remote-host row.
  4. Fill Host / Device URL, Port, OS, Username, Auth Type, and Password if needed.
  5. Select that row and click Connect && Load Interfaces.
  6. If Packetra connects successfully, the lower tree is populated with Remote Host / Interface entries.
  7. Check the Show boxes for the interfaces you want visible in the main interface list.
  8. Click OK or close the dialog in a way that saves settings.
  9. Back in the interface picker / capture options screen, the selected remote interfaces appear as normal selectable capture targets.
  10. Select the desired remote interface.
  11. If needed, assign a capture filter to that interface.
  12. Start capture. Packetra then opens the SSH capture stream and ingests packets locally.

Important implementation detail: Packetra stores remote host rows and the per-interface Show state, so you do not need to rediscover everything every time unless the server or interface list changed.

7. What Packetra sends to the Linux host

Understanding the exact backend behavior helps troubleshooting:

  • Interface list command: tcpdump -D
  • Capture command base: tcpdump -n -s 0 -i 'iface' -U -w -
  • Non-promiscuous mode adds: -p
  • Capture filter appends directly as a BPF expression

Examples:

tcpdump -n -s 0 -i 'eth0' -U -w -
tcpdump -n -s 0 -i 'eth0' -U -w - -p
tcpdump -n -s 0 -i 'eth0' -U -w - port 443
tcpdump -n -s 0 -i 'ens160' -U -w - host 10.10.10.25 and tcp port 8443

8. Recommended Linux validation sequence before using Packetra

ssh user@remote-host
which tcpdump
tcpdump -D
tcpdump -n -s 0 -i eth0 -c 5
exit

If these commands work manually, Packetra is very likely to work as well once the same host, username, and auth method are entered correctly.

9. How to keep the server side always available in the background

Packetra does not require a Linux “agent” service for remote capture. What must stay available in the background is mainly the SSH service, and optionally your admin/session tooling.

Keep SSH enabled permanently

sudo systemctl enable ssh
sudo systemctl start ssh

or, on systems using sshd:

sudo systemctl enable sshd
sudo systemctl start sshd

Verify SSH survives reboot

sudo systemctl is-enabled ssh
sudo systemctl is-active ssh

or:

sudo systemctl is-enabled sshd
sudo systemctl is-active sshd

If you want your own maintenance shell to keep running

This is not required for Packetra itself, but useful during setup, testing, or long troubleshooting sessions:

tmux new -s packetra-setup
screen -S packetra-setup
nohup tcpdump -D > /tmp/packetra_ifaces.txt 2>&1 &

If you want tcpdump capability to remain available after reboot

Capabilities may need to be re-applied depending on how your environment manages binaries. Re-check after package upgrades.

getcap $(command -v tcpdump)
sudo setcap cap_net_raw,cap_net_admin=eip $(command -v tcpdump)

10. Authentication choices

  • Key / agent auth: preferred for stable SSH-based remote work.
  • Password auth: works when enabled, but only if the password field is filled and the server permits password login.

Packetra reuses SSH connections when it can, so repeated interface loads are usually faster after the first successful connection.

11. Start-of-work checks and end-of-work checks

Start-of-work checks

  1. Confirm the host is reachable with manual SSH.
  2. Confirm tcpdump exists.
  3. Confirm tcpdump -D lists interfaces.
  4. Confirm the chosen SSH user has enough permission to capture or list interfaces.
  5. Confirm Packetra can load interfaces through Connect && Load Interfaces.

End-of-work checks

  1. Stop the live capture in Packetra before closing the analysis session.
  2. Save the capture locally if it is needed for later analysis.
  3. If you changed Linux capture capabilities temporarily, decide whether they should remain in place.
  4. If you opened extra validation shells with tmux, screen, or background jobs, clean them up if they are no longer needed.

13. Windows Remote Capture via Agent Service

Windows does not natively provide a tool equivalent to tcpdump that can easily stream raw packets over an SSH connection without external dependencies. Packetra therefore uses a Windows Agent MSI that prepares the machine and registers a service for remote capture.

Installation Steps:

  1. On the Packetra Host: Open Capture > Manage Interfaces, go to the Remote Interfaces tab, and click Install Agent.
  2. Save the packetra-remote-agent.zip file. It contains the PacketraAgent.msi installer.
  3. Transfer packetra-remote-agent.zip to your target Windows machine.
  4. Extract the ZIP and run the PacketraAgent.msi installer on that machine.
  5. The MSI now checks required prerequisites first. If OpenSSH Client/Server are missing, it installs them. If they already exist, it reuses them.
  6. The MSI then checks Npcap. If Npcap is missing, the installer downloads the official Npcap installer and runs it. If Npcap is already installed, it is skipped.
  7. After prerequisites are ready, the MSI creates or updates the PacketraAgent Windows service and starts it automatically.
  8. If the agent is already installed, running the MSI again enters standard Windows Installer maintenance mode so you can choose Repair or Remove.

Validation & Checking the Service:

Once installed, you do not need to manually run the capture agent executable. The service runs automatically in the background and starts on boot.

  • Open services.msc and confirm PacketraAgent exists, is Running, and uses Automatic startup.
  • Run Get-Service PacketraAgent in PowerShell and confirm the service is present.
  • Run where.exe RemoteCaptureAgent.exe and confirm the executable resolves to the PacketraAgent install directory.
  • Run Get-Service sshd and confirm OpenSSH Server is present and running.
  • Run Get-Service npcap -ErrorAction SilentlyContinue and confirm Npcap is installed.

Recommended PowerShell check block:

Get-Service PacketraAgent -ErrorAction SilentlyContinue
Get-Service sshd -ErrorAction SilentlyContinue
Get-Service npcap -ErrorAction SilentlyContinue
where.exe RemoteCaptureAgent.exe
where.exe ssh.exe

If the MSI fails while preparing prerequisites or the service, check the bootstrap log too:

Get-Content "$env:TEMP\PacketraAgent-bootstrap.log" -ErrorAction SilentlyContinue

Connecting from Packetra:

After verifying the service, return to Packetra, add the Windows Host IP/credentials in the Remote Interfaces tab, select OS as windows, and click Connect & Load Interfaces. Packetra only needs SSH connectivity from the server side at this point; the client machine should already have everything else prepared by the MSI.

14. Remove the Windows Agent completely

The installed service name is PacketraAgent. Depending on how it was installed, the executable may live under C:\Program Files\PacketraAgent or a user-local app directory.

Step 1: Find the exact Agent path and current service state

Open PowerShell (Run as Administrator) and run:

$svc = Get-ItemProperty 'HKLM:\SYSTEM\CurrentControlSet\Services\PacketraAgent' -ErrorAction SilentlyContinue
$img = $svc.ImagePath
if ($img) { Write-Host "[+] Service ImagePath: $img" -ForegroundColor Green }
Get-Service PacketraAgent -ErrorAction SilentlyContinue
where.exe RemoteCaptureAgent.exe

Step 2: Use MSI maintenance first

If you still have the same PacketraAgent.msi installer, run it again first. Windows Installer should offer Repair or Remove. Choose Remove for the cleanest supported uninstall path.

Step 3: Manual cleanup if the service or files remain

If the MSI remove path leaves remnants behind, stop and remove the service manually:

sc.exe stop PacketraAgent
sc.exe delete PacketraAgent
taskkill /f /im RemoteCaptureAgent.exe

Then delete the remaining install directory. If the folder is inside C:\Program Files\..., use an Administrator prompt or File Explorer:

  • Method A (File Explorer): Navigate to C:\Program Files, right-click the Agent folder, and select Delete. Click Continue when prompted for Administrator permission.
  • Method B (PowerShell as Admin): Replace C:\path\to\AgentFolder with the real path you found:
Remove-Item -Recurse -Force "C:\path\to\AgentFolder"

Step 4: Optional prerequisite cleanup

Normally you can leave OpenSSH and Npcap installed, especially if other tools use them. If you really want to remove them too, do so separately and only if you know they are not needed elsewhere:

Remove-WindowsCapability -Online -Name OpenSSH.Server~~~~0.0.1.0
Remove-WindowsCapability -Online -Name OpenSSH.Client~~~~0.0.1.0

Npcap should usually be removed through Apps & Features or the Npcap uninstaller instead of deleting files manually.

Verify removal

After cleanup, run the checks again:

Get-Service PacketraAgent -ErrorAction SilentlyContinue
where.exe RemoteCaptureAgent.exe

The service should no longer appear and the executable path should no longer resolve.
