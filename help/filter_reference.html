Display Filter Reference

This page documents the display filter engine currently implemented in Packetra and every major in-app way a user can create, apply, combine, copy, or reuse a display filter. Every example here is intended for the main display filter system, not capture-filter fields.

3. Step-by-step display-filter entry paths

Main filter bar

  1. Locate the filter bar above the packet list.
  2. Type an expression.
  3. Press Enter or click the arrow button.
  4. Use the x button to clear it.

History and autocomplete

  1. Click the trailing drop-down icon inside the filter input to open recent filters.
  2. Choose an older expression to reapply it immediately.
  3. While typing, use autocomplete suggestions for protocols, fields, and operators.

Drag and drop from packet list

The packet list supports dragging from these columns only:

  • Source → creates src == "value"
  • Destination → creates dst == "value"
  • Protocol → creates protocol == "value"
  • Length → creates frame.len == number
  1. Left-click one of those cells.
  2. Drag it onto the display filter input.
  3. Drop it into the filter bar.
  4. The filter bar is populated with the generated expression.

Packet list right-click context menu

  1. Right-click a packet-list cell.
  2. Open Apply as a Filter.
  3. Choose one of:
    • Selected
    • Not Selected
    • ... and Selected
    • ... or Selected
    • ... and not Selected
    • ... or not Selected

Packet details right-click or Analyze menu

  1. Select a field in the packet details tree.
  2. Either right-click and use Apply as a Filter, or open Analyze > Apply as Filter.
  3. Packetra builds a strict detail filter based on the selected node such as detail.pair, detail.key, or detail.title.

Expression builder

  1. Open Analyze > Display Filter Expression....
  2. Choose a field.
  3. Choose a relation.
  4. Enter a value.
  5. Choose whether to replace, AND, OR, AND NOT, or OR NOT with the current filter.
  6. Use Copy, Insert into Filter Bar, Apply, or OK.

Macros

  1. Open Analyze > Display Filter Macros....
  2. Create a macro name and expression.
  3. Save it.
  4. Use that macro name later inside a display-filter expression where macro expansion is supported by Packetra’s filter input flow.

Conversation filter and Follow Stream

  1. Select a packet.
  2. Use Analyze > Conversation Filter to build a TCP, UDP, IPv4, IPv6, or Ethernet conversation expression.
  3. Or open Follow and use Filter This Stream / Filter Out This Stream.

6. Operator behavior notes

  • contains is case-insensitive substring matching.
  • Numeric operators are numeric when both sides can be parsed as numbers; otherwise comparison falls back to case-insensitive text comparison.
  • A bare field without an operator acts as a boolean presence test. Example: http.request is true when the engine recognizes an HTTP request.
  • A bare protocol alias such as tcp or dns triggers protocol-aware matching logic, not just simple text matching.

8. Statistics and visual tools that can push filters back into the main filter bar

  • Protocol Hierarchy: selected protocol can be applied as a display filter.
  • Endpoints: selected endpoint rows can be converted into host/address filters.
  • Conversations: selected rows can push conversation expressions.
  • Flow Graph: selected rows can apply an associated filter expression.
  • HTTP Statistics: common status-class and request/response filters can be pushed into the filter bar.
  • IPv4 Statistics / IPv6 Statistics: selected address entries can become address-based filters.
  • Network Topology Graph: selected nodes or edges can produce address/protocol filter expressions through Apply Filter or Copy Filter.

9. What is not implemented in this engine

  • Wireshark-style fields that are not explicitly listed above are not guaranteed to work.
  • Expressions such as frame contains "text" are not part of the documented implemented field set unless a corresponding field exists. Prefer detail contains ... or a concrete field name.
  • Advanced Wireshark-only helpers like tcp.analysis.retransmission are not implemented by this display-filter engine unless Packetra exposes equivalent fields.

10. Debugging a filter that returns nothing

  1. Start with a protocol token such as tcp or dns.
  2. Add a single IP field next, for example tcp and ip.addr == 10.0.0.5.
  3. Add ports after that, for example tcp and ip.addr == 10.0.0.5 and tcp.port == 443.
  4. Only then add string conditions like contains.
  5. If a detail-based condition fails, click a matching packet first and verify the exact details-tree text that Packetra is generating.
