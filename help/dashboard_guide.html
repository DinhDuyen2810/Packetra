Dashboard Guide

This guide explains the dashboard system in detail: what each template is good at, what each widget means, which data source powers it, and how to combine dashboards with filters and packet-level inspection.

1. Dashboard concepts

ConceptMeaning
TemplateA prebuilt dashboard layout such as Network Overview or DNS Analysis.
WidgetA single visual block: metric, table, bar chart, line chart, donut, topology, and so on.
Data sourceThe backing dataset used by a widget, such as packets, endpoints, conversations, protocol_stats, dns_queries, or http_requests.
QueryThe widget’s filter, grouping, metrics, sorting, limit, and time bucketing settings.
VisualizationHow the query result is rendered: metric, table, bar, line, area, scatter, radar, treemap, sunburst, pie, donut, histogram, heatmap, or topology.
Data scopeDashboards typically operate on the currently displayed packets, so your current display filter matters.

3. Available visualization types in the current build

The dashboard system registers these visualization types: metric, table, bar, horizontal_bar, line, area, scatter, radar, treemap, sunburst, pie, donut, histogram, heatmap, and topology.

  • Use metric when you want one number such as total packets or distinct DNS queries.
  • Use table when you need exact rows for drill-down or reporting.
  • Use bar or donut when comparing categories such as protocols, endpoints, or ports.
  • Use line when time is the primary dimension.
  • Use topology when the relationship between nodes matters more than raw count tables.

4. Template-by-template guide

Network Overview

  • Total Packets (`packets`, `metric`): immediate sense of volume.
  • Total Bytes (`packets`, `metric`): useful for comparing noisy captures that do not necessarily have many packets.
  • Protocol Distribution (`protocol_stats`, `donut`): the fastest way to see whether the capture is DNS-heavy, HTTP-heavy, TLS-heavy, or dominated by one unexpected protocol.
  • Packets Over Time (`packets`, `line`): shows burstiness, steady flow, or silent gaps.
  • Top Endpoints (`endpoints`, `table`): shows who is talking the most.

Best use: first dashboard to open on an unfamiliar capture. Combine it with a broad display filter, then narrow further once you know the dominant hosts and protocols.

Timeline Analysis

  • Packets Over Time (`packets`, `line`): packet-rate rhythm.
  • Bytes Over Time (`packets`, `line`): volume rhythm; helps distinguish many-small-packet traffic from fewer large transfers.
  • Top Conversations (`conversations`, `bar`): which conversations dominate during the displayed time scope.
  • Protocol Mix (`protocol_stats`, `donut`): what kinds of traffic are present in the same time-filtered slice.

Best use: incidents with spikes, bursts, timing anomalies, or “something happened around this minute” cases.

DNS Analysis

  • Distinct Queries (`dns_queries`, `metric`): quick count of unique DNS questions.
  • Top DNS Queries (`dns_queries`, `bar`): domains with the most attention.
  • DNS Query Table (`dns_queries`, `table`): exact domain list and counts.
  • DNS Response Time (`dns_queries`, `table`): sort by latency to spot slow or failing lookups.

Best use: name-resolution issues, suspicious domains, repeated retries, and malware-style beacon domain patterns.

HTTP/TLS Analysis

  • HTTP Messages (`http_requests`, `metric`): whether actual HTTP content is present in the visible scope.
  • HTTP vs SSL (`packets`, `donut`): rough split between plain HTTP and encrypted traffic as modeled by the template.
  • HTTP Request Table (`http_requests`, `table`): method, URI, status, latency, and peer addresses.
  • Largest SSL Packets (`packets`, `table`): useful when TLS traffic is dominant and you want large-transfer clues.

Best use: web troubleshooting, API tracing, login failures, or TLS-heavy application flows.

Endpoint Activity

  • Top Talkers by Bytes (`endpoints`, `bar`): who moves the most data.
  • Top Talkers by Packets (`endpoints`, `bar`): who sends the most units of traffic.
  • Endpoint Table (`endpoints`, `table`): exact address, packet, byte, and protocol participation summary.

Best use: identifying noisy clients, busy servers, or one host dominating the conversation graph.

Protocol Analysis

  • Top Protocols (`protocol_stats`, `bar`): protocol ranking by packet count.
  • Protocol Bytes (`protocol_stats`, `donut`): protocol ranking by byte volume.
  • Top DNS Queries (`dns_queries`, `table`): lets DNS stand out even in a broader protocol review.
  • HTTP Messages (`http_requests`, `table`): gives application-level context inside the same dashboard.

Best use: multi-protocol captures where you need a fast traffic taxonomy before drilling into one protocol family.

Security Investigation

  • Top Destination Ports (`packets`, `bar`): surfaces hot ports and possible scan focus areas.
  • Slow DNS Responses (`dns_queries`, `table`): useful for suspicious or failing name-resolution paths.
  • Largest Conversations (`conversations`, `table`): exposes the heaviest exchanges that may deserve packet-level review.

Best use: suspicious activity triage, service discovery patterns, noisy traffic, or broad security review.

Topology View

  • Conversation Bytes (`conversations`, `bar`): shows the heaviest relationships.
  • Protocol Distribution (`protocol_stats`, `donut`): explains what kinds of relationships dominate the graph.
  • Top Conversations (`conversations`, `table`): gives exact pairs to validate the graph visually.

Best use: understanding node-to-node structure, peer relationships, and major traffic pathways.

5. How to combine dashboards effectively

  1. Start with Network Overview to understand the broad shape.
  2. Move to Timeline Analysis if the issue is time-dependent.
  3. Move to Protocol Analysis, DNS Analysis, or HTTP/TLS Analysis once one protocol family becomes suspicious.
  4. Open Endpoint Activity or Topology View if you need to understand relationships between hosts.
  5. Use Security Investigation when the question becomes “which pattern looks dangerous or abnormal?”

In practice the strongest combination is often:

Network Overview
-> Endpoint Activity
-> Timeline Analysis
-> Protocol-specific dashboard
-> Packet list / packet details for proof

6. How display filters interact with dashboards

  • Most dashboard templates operate on the currently displayed packet scope.
  • This means your active display filter acts like a pre-filter for the dashboard.
  • Example: if you apply dns first and then open Network Overview, the totals and charts reflect only displayed DNS packets, not the entire original capture.

Recommended pattern:

1. Apply a broad display filter, e.g. tcp.port == 443
2. Open Network Overview
3. If needed, refine to one host, e.g. tcp.port == 443 and ip.addr == 10.0.0.12
4. Re-open or refresh dashboard interpretation
5. Drill into specific packets

8. Importing dashboards and creating JSON files manually

The Import Dashboard button is active in the current build and loads a .json file through the dashboard repository. A real round-trip test confirms the flow works: export a dashboard to JSON, import it again, and the application saves it into the user dashboard store. If the imported dashboard_id already exists, Packetra automatically generates a new dashboard ID so the import does not overwrite the existing one.

What the import button expects

  1. Open the Dashboard home page.
  2. Click Import Dashboard.
  3. Select a file with the extension .json.
  4. Packetra reads the whole file as text and tries to convert it into a Dashboard object.
  5. If the JSON matches the schema, the dashboard is saved and appears in My Dashboards.
  6. If the JSON is malformed or missing required structure, import fails and the dialog shows an error or failure message.

Fastest and safest way to create an import file

If you do not already know the schema, use this workflow first:

  1. Create a dashboard inside Packetra.
  2. Add one or more widgets and save it.
  3. Open the dashboard card menu and choose Export JSON.
  4. Use that exported file as your base template.
  5. Duplicate it, then edit only the fields you need to change.

This is the most reliable method because your starting file already matches the exact schema used by the running build.

If you want a ready-made reference file, download sample_dashboard_import.json. You can open it in any text editor, inspect the structure, then change the name, IDs, filters, widgets, and layout before importing it into Packetra.

Dashboard JSON structure

The imported file is one JSON object representing a full dashboard. These top-level fields matter the most:

FieldRequiredMeaning
schema_versionYesDashboard schema version. Current dashboards use 1.
dashboard_idYesUnique dashboard identifier. Use a UUID-like string. If it already exists, Packetra generates a new one during import.
nameYesThe dashboard name shown in the gallery.
descriptionNoOptional text shown under the dashboard name.
is_templateYesCan be present as either true or false, but imported dashboards are forced to user dashboards internally.
created_atYesISO timestamp string.
updated_atYesISO timestamp string.
layoutYesDashboard grid settings, usually columns and row_height.
widgetsYesArray of widget objects. An empty list is valid, but the dashboard will open with no widgets.
global_filterNoOptional dashboard-wide query filter string.
time_rangeNoOptional time-range object.
data_scopeNoUsually displayed_packets or all_packets.
tagsNoOptional list of tags.
thumbnailNoOptional thumbnail metadata.

Widget object structure

Each widget inside widgets must include these blocks:

FieldRequiredMeaning
idYesUnique widget ID.
titleYesWidget title shown in the dashboard.
data_sourceYesDataset used by the widget, such as packets, endpoints, conversations, protocol_stats, dns_queries, or http_requests.
queryYesWidget query block.
visualizationYesChart or table configuration block.
layoutYesWidget position and size in the grid: x, y, w, h.
descriptionNoOptional text shown below the widget.
styleNoOptional style and display settings such as chart/table display mode and widget-specific editor settings.

Query block structure

FieldMeaning
filterOptional widget-level filter string.
group_byOptional list of grouping fields.
metricsOptional list of metrics. Each metric uses type, field, and as.
sortOptional list of sort rules. Each sort uses field and direction.
limitOptional integer result limit.
time_bucketOptional bucket size such as 1s, 5s, 1m, or 1h.
columnsOptional table column list.

For metric aliases, JSON uses the key as. Internally the Python model maps it to as_, but you should write "as": "packets" in the JSON file.

Visualization block structure

The visualization object always needs at least type. Depending on chart type, you may also use:

  • x_field and y_field for XY charts such as line, area, and scatter.
  • category_field and value_field for category/value charts such as bar, pie, donut, radar, treemap, and sunburst.
  • series_field when the chart uses series grouping or split colors.
  • show_legend and show_labels for display behavior.

Minimal import example

This is a compact example that should be understandable even if you are creating your first import file:

{
  "schema_version": 1,
  "dashboard_id": "11111111-2222-3333-4444-555555555555",
  "name": "My Imported Dashboard",
  "description": "Simple imported dashboard example",
  "is_template": false,
  "created_at": "2026-06-17T08:00:00",
  "updated_at": "2026-06-17T08:00:00",
  "layout": {
    "columns": 12,
    "row_height": 80
  },
  "widgets": [
    {
      "id": "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee",
      "title": "Top Protocols",
      "data_source": "protocol_stats",
      "query": {
        "group_by": ["protocol"],
        "metrics": [
          { "type": "count", "field": "*", "as": "packets" }
        ],
        "sort": [
          { "field": "packets", "direction": "desc" }
        ],
        "limit": 10
      },
      "visualization": {
        "type": "bar",
        "category_field": "protocol",
        "value_field": "packets",
        "show_legend": false,
        "show_labels": true
      },
      "layout": {
        "x": 0,
        "y": 0,
        "w": 6,
        "h": 4
      },
      "description": "Top protocols by packet count",
      "style": {
        "display_mode": "chart"
      }
    }
  ],
  "global_filter": "tcp or udp",
  "data_scope": "displayed_packets",
  "tags": ["import", "example"]
}

How to build a JSON file from zero if you know nothing yet

  1. Create a new plain text file.
  2. Save it with a .json extension, for example my_dashboard.json.
  3. If you prefer not to start from a blank file, download sample_dashboard_import.json and edit that instead.
  4. Paste the minimal example above.
  5. Change name to the dashboard title you want.
  6. Change each ID string so your dashboard and widget IDs are unique. UUID-format strings are safest.
  7. Keep schema_version at 1.
  8. Keep layout.columns at 12 unless you know you want another grid width.
  9. Start with one widget first. Do not try to build ten widgets until one working widget imports correctly.
  10. Choose a valid data_source.
  11. Choose a visualization type that matches that source and query result.
  12. Save the file.
  13. Import it through the Dashboard page.

Recommended beginner workflow for creating larger imports

  1. Create a dashboard in the UI.
  2. Add one widget and configure it until it renders correctly.
  3. Save the dashboard.
  4. Export it as JSON.
  5. Use that export as your schema reference.
  6. Duplicate the widget block and only then start editing titles, positions, filters, metrics, and visualization fields.
  7. Re-import the new file under a different dashboard name for testing.

How to place widgets in the grid

  • x: starting column, usually from 0 to 11.
  • y: starting row, usually 0 for the first row.
  • w: width in columns.
  • h: height in rows.

Example: a widget with "x": 0, "y": 0, "w": 6, "h": 4 starts in the top-left corner and takes half the dashboard width when the dashboard uses 12 columns.

Important compatibility rules

  • A table widget should usually provide columns in the query or at least return fields that make sense in a table.
  • A bar, pie, donut, radar, treemap, or sunburst chart usually needs category/value data.
  • A line, area, or scatter chart needs axis-compatible fields, especially a numeric value field.
  • If your chosen fields do not exist in the selected data source, the widget may import but show empty or non-plottable output.
  • Import validates the JSON structure, but it does not guarantee your chosen field names will produce useful data at runtime.

Common mistakes that cause confusion

  • The file is valid JSON but uses field names that the chosen data source never returns.
  • The widget imports but the chart says No data or No plottable data because the query result is empty.
  • The widget uses a chart type that does not match the query shape.
  • The dashboard imports with the same name as another dashboard, which is allowed, but can confuse users if you do not rename it clearly.
  • The JSON is edited manually and a comma, quote, or bracket is missing.

What happens after import

  • The imported dashboard is stored as a user dashboard.
  • is_template is forced to false internally.
  • The dashboard is added to the user dashboard repository file.
  • You can then open it, rename it, duplicate it, export it again, or delete it like any other user dashboard.

9. What dashboards do not replace

  • Dashboards do not replace packet-level verification.
  • Dashboards do not guarantee causal explanation; they show shape, concentration, and ranking.
  • When a widget looks suspicious, the next step should still be a filter plus packet inspection.
